### H1. `ErrorReporter.print_errors` caret offset is wrong by the span's `start` offset

**File:** `src/error/mod.rs` L242-249

**Detail:** The caret underline is computed as `error.span.end.saturating_sub(error.span.start)` characters of `^`, but it is indented from the pipe by a fixed `spaces` that aligns only with the line-number prefix. The carets start at column 0 of the source line, not at `error.span.column`. The caret therefore underlines the wrong range: starting from the left edge of the line instead of the actual span column.

```rust

let carets = "^".repeat(error.span.end.saturating_sub(error.span.start).max(1));
eprintln!("{}  \x1b[32m{}\x1b[0m", spaces, carets);
```

**Impact:** Anyone using the CLI compiler with multi-error output sees carets that don't line up with the actual error location. Misleading diagnostic presentation.

**Reproduction:** Create a `.cell` file with a type error that triggers `ErrorReporter::print_errors()`. The caret will start at the beginning of the line rather than at the span column.

### F4. **MEDIUM — Silent emit suppression on offset overflow**

**Files:** `src/codegen/frame.rs:122-124`, `src/codegen/mod.rs:511-518`

### F-1 [High] `&mut T` silently accepted where `&T` is expected -- aliasing check incomplete

**Location:** `types/mod.rs:5708-5713`  
**Invariant violated:** A function declared with `&T` parameter can receive `&mut T` from the caller. The callee treats it as read-only, but the caller retains mutable access. The `reject_duplicate_mutable_reference_call_roots` guard only flags when the same root appears multiple times with at least one `MutRef` in the **expected** signature. It does not catch the case where the caller passes the same mutable reference root as both a `&T` argument and continues to use it as `&mut T` after the call.

**Call-site flow:**
1. `infer_expr` infers the arg has type `MutRef(T)` 
2. `validate_call_args` calls `call_argument_type_compatible(&T, &mut T)` → `true`
3. `reject_duplicate_mutable_reference_call_roots` skips because `actual` is `MutRef` but `expected` is `Ref` → `participates_in_mutable_alias = false`
4. The root is recorded with `participates = false`, so a second occurrence wouldn't be caught either

**Example:**
```cellscript
fn reader(x: &MyStruct) { ... }
fn writer(x: &mut MyStruct) { ... }

// Both called with the same &mut root:
let r: &mut MyStruct = &mut data;
reader(r);  // coerces &mut to &, passes
writer(r);  // passes &mut directly
// No error -- the &mut root was recorded once with participates=false, once with true
```

**Uncertainty:** This may be intentional CellScript design. Rust also allows `&mut T` to coerce to `&T` at call sites, but Rust's borrow checker also prevents the caller from using `&mut` while `&` is live. CellScript lacks a full borrow checker, making this potentially unsound.

### F-2 [High] `Expr::If` does not merge type refinements from branches back to parent

**Location:** `types/mod.rs:3638-3648`  
**Invariant violated:** Type refinements (e.g., `Vec` becoming `Vec<u8>` via `push`) that occur inside an if-expression's branches are lost because `merge_existing_type_refinements_from` is never called.

`merge_existing_type_refinements_from` is called only from:
- `Stmt::For` path (line 3104)
- `Stmt::While` path (line 3116)

It is **not** called from:
- `Stmt::If` path
- `Expr::If` path  
- `Expr::Block` path

For `Expr::If`, after evaluating both branches in child environments, only linear states are merged. Any type mutations to parent-scoped variables are silently dropped.

**Example:**
```cellscript
let v = if cond {
    let mut v = Vec::new();
    v.push(42u64);
    v  // type is Vec<u64> in child env
} else {
    Vec::new()  // type is Vec (untyped) in child env
};
// Result type: Vec (untyped), not Vec<u64>
```
