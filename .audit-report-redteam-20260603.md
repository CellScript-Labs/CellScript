## CRITICAL

### RT-C1. Fatal stack overflow on deeply nested parentheses

- **ID:** RT-C1
- **File:** `src/parser/mod.rs` L5, L49-61
- **Description:** The parser's recursion guard (`MAX_PARSE_RECURSION_DEPTH = 128`) uses `with_recursion_guard` which checks depth *before* incrementing. However, the `parse_primary` path for parenthesized expressions (`parse_group`) calls `self.parse_expr()` which calls `with_recursion_guard` — the guard correctly fires at depth 128 and returns `Err`. The problem is that the test `deeply_nested_parentheses_in_parser` at depth 256 causes a **fatal process abort** (`SIGABRT: stack overflow`) instead of a graceful error. This means the recursion guard is being bypassed or the depth accounting is wrong in some edge case.
- **Impact:** A malicious or buggy CellScript source file with deeply nested parentheses causes the compiler process to abort with no error message. This is a denial-of-service vector for any service that compiles untrusted CellScript code (e.g., a blockchain node compiling deployment scripts).
- **Test evidence:** `cargo test --test adversarial_audit deeply_nested_parentheses_in_parser` → `thread '...' has overflowed its stack\nfatal runtime error: stack overflow, aborting`
- **Suggested fix:**
  1. Verify that `with_recursion_guard` is applied to every recursive call path, including `parse_group` / parenthesized expressions.
  2. Increase the Rust test thread stack size via `#[test]` attribute `#[ignore]` or set `RUST_MIN_STACK` environment variable, and add a `stacker` dependency to guard all recursive compiler paths.
  3. Consider converting deeply recursive AST traversal (type checker `infer_expr`, IR `lower_expr`) to use an explicit stack/loop with a depth limit.

## MEDIUM

### RT-M1. No recursion depth guard in type checker `infer_expr` or IR `lower_expr`

- **ID:** RT-M1
- **File:** `src/types/mod.rs` L3515 (infer_expr), `src/ir/mod.rs` L2049 (lower_expr)
- **Description:** Both `infer_expr` and `lower_expr` recursively walk the AST for binary expressions. A left-associative chain like `1+1+1+...+1` (N terms) creates a left-nested AST `(((1+1)+1)+1)+...` with recursion depth N. The parser uses iterative precedence climbing (no recursion cost for binary ops), but the type checker and IR each recurse to depth N.

  Testing showed that depths up to **50,000** binary additions compile successfully without stack overflow. The stack frames are small (~200 bytes each with Box-dereferenced operands), and the default Rust test stack is ~8MB, so ~40K recursive calls fit. However, a combined depth (type checker N + IR N + other recursive traversals) could theoretically overflow on smaller stacks (e.g., in embedded or WASM-compiled contexts).

  The production `cellc` binary likely has a larger stack, but there is **no explicit depth limit** to prevent abuse.
- **Impact:** An adversarial source with extremely deep expression chains (100K+ terms) could stack-overflow the compiler in contexts with limited stack size. On the default CLI, the risk is low but unbounded.
- **Suggested fix:** Add an explicit depth counter to `infer_expr` and `lower_expr` (similar to the parser's `recursion_depth`), and emit a compile error if it exceeds a configurable limit (e.g., 10,000). This also bounds compile time for adversarial inputs.

### RT-M2. `encode_u_type` silently truncates `i64` to `i32` without range check

- **ID:** RT-M2
- **File:** `src/codegen/assembler.rs` L1886-1887
- **Description:** The `encode_u_type` function encodes RISC-V U-type instructions (used for `lui` and `auipc`). It takes `imm: i64` and truncates via `imm as i32 as u32`, masking to 20 bits. Unlike `encode_signed_bits` (used for I/J/B/S-type encodings), it does NOT check whether the value fits before truncating.

  In practice, `encode_u_type` is only called from `encode_li_sequence`, `encode_address_sequence`, `encode_call_sequence`, and `encode_long_jump_sequence`, all of which use `split_hi_lo` which already validates the range via `li_fits_lui_addi_rv64`. So the truncation is currently safe due to upstream guards.
- **Impact:** No current bug, but the function is a latent trap. If a new call site uses `encode_u_type` without the `split_hi_lo` guard, an out-of-range immediate would silently encode the wrong value into the instruction, producing an incorrect jump address or loaded constant. This would silently corrupt on-chain contract bytecode.
- **Suggested fix:** Add a range check to `encode_u_type` itself, matching the pattern in `encode_signed_bits`:
  ```rust
  fn encode_u_type(opcode: u32, rd: u8, imm: i64) -> Result<u32> {
      if !signed_bits_fit(imm, 20) {
          return Err(...);
      }
      Ok((((imm as i32 as u32) & 0x000f_ffff) << 12) | ((rd as u32) << 7) | opcode)
  }
  ```

## LOW

### RT-L1. `expected_elements as u32` truncation in constructed vector verification

- **ID:** RT-L1
- **File:** `src/codegen/cell_ops.rs` L1558
- **Description:** `(expected_elements as u32).to_le_bytes()` converts the expected element count to a u32 for byte-level comparison in generated assembly. If `expected_elements > u32::MAX`, this silently truncates.
- **Impact:** In practice, CKB cells have a maximum data size of 64KB (or ~1MB with newer proposals), so element counts cannot plausibly exceed u32::MAX. This is a theoretical truncation only.
- **Suggested fix:** Add a debug assertion or `try_from` conversion.

## INFO

### RT-I1. Codegen uses iterative (non-recursive) instruction emission

- **ID:** RT-I1
- **File:** `src/codegen/mod.rs` L405-460 (`generate`), L1485-1530 (`generate_body`)
- **Description:** The codegen backend iterates over IR blocks and instructions using `for` loops. It does NOT recursively traverse the IR tree. Each `IrInstruction` is handled by pattern matching in a flat dispatch. This means the codegen cannot stack overflow from deeply nested ASTs regardless of input depth.
- **Impact:** Positive. Stack overflow risk is bounded to the type checker and IR lowering stages.

### RT-I2. Parser binary expression parsing is iterative (Pratt/precedence climbing)

- **ID:** RT-I2
- **File:** `src/parser/mod.rs` L2024-2194 (`parse_or` through `parse_factor`)
- **Description:** All binary operator precedence levels use `loop { ... break }` patterns, not recursive descent. A chain of N binary operators parses in O(N) time with O(1) stack depth. This is correct and efficient.
- **Impact:** Positive. Binary expression chains do not contribute to parser stack depth.
